# Compile 自動処理ツール 企画書 (v1)

作成日: 2026-06-12

## 1. 背景と目的

現状のツール群はいずれも「手動プレイ補助」:

| ツール | 役割 | 効果処理 |
|---|---|---|
| solo-play.html | 1端末プレイ + 簡易AI | 手動(プレイヤーが効果文を読んで適用) |
| room-play.html | Firebase オンライン対戦 | 手動 |
| control-tracker.html | コントロール管理 | 手動 |

本企画は **カード効果・ルールを完全自動で解決するデジタル版 Compile** を作る。
プレイヤーは「カードを選ぶ」「対象を選ぶ」だけで、ドロー・反転・移動・削除・
コンパイル判定・コントロール判定・キャッシュ/リフレッシュがすべて自動進行する。

## 2. スコープ

- 対象セット: Main 1 (12プロトコル) + Aux 1 (3プロトコル) = 90枚
  - 効果スロット 104 / ユニーク効果文 89 (全て手動でデータ化可能な規模)
- 対象モード(段階導入):
  1. ソロ (vs AI)
  2. パス&プレイ (1端末2人)
  3. オンライン対戦 (room-play 統合)

## 3. アーキテクチャ方針

リポジトリの流儀(ビルドなし・単一HTML)は維持しつつ、エンジンだけ分離する。

```
data/cards.json      … 表示用カードデータ(既存)
data/effects.json    … 効果DSL(新規・カードIDで紐付け)
engine.js            … ルールエンジン(UI非依存・純粋関数)
auto-play.html       … 自動処理版UI(solo-play のUI資産を流用)
room-play.html       … 将来エンジンを組み込み(Phase 5)
test/engine.test.mjs … node --test によるエンジン単体テスト
```

### 3.1 効果DSL

自然文をパースせず、89種の効果を**宣言的JSONで手動定義**する。例:

```json
"DARKNESS_1": {
  "middle": [
    { "op": "draw", "count": 3 },
    { "op": "shift", "target": { "owner": "opp", "state": "covered" }, "optional": false }
  ]
},
"DARKNESS_3": {
  "upper": { "type": "static", "op": "setValue", "value": 4,
             "filter": { "zone": "thisStack", "facing": "down" } }
}
```

- `middle` = プレイ時コマンド(逐次実行)
- `upper` = 常在効果(値修正・制約) → 値計算時に毎回評価
- `lower` = トリガー(開始時/終了時/〜したとき) → イベントフックで発火

### 3.2 エンジン (engine.js)

- `apply(state, action) → { state', requests[], log[] }` の純粋関数型ステートマシン
- 対象選択・任意効果は **request(選択要求)** としてUIに返し、UIが
  `{ type:"choose", ... }` アクションで応答する中断/再開モデル
- 乱数はシード付き(オンライン同期・テスト再現性のため)
- 自動処理: ターン進行 / フェイズ / トリガー解決順 / コンパイル条件
  (ライン値10以上かつ相手超過) / 勝利判定 / コントロール / リフレッシュ

### 3.3 UI (auto-play.html)

- solo-play.html のビジュアル(ライン・チップ・ドラッグ)を流用
- 合法手のハイライト(エンジンが legal moves を列挙)
- 選択要求モーダル(対象カード選択・任意効果の Yes/No)
- アクションログ(自動処理の透明性確保)・アンドゥ(state スナップショット)

## 4. フェーズ計画

| Phase | 内容 | 成果物 |
|---|---|---|
| 0 | ルール仕様化: Rulesheet PDF + Codex からターン構造・タイミング・用語を文書化 | docs/rules-spec.md |
| 1 | 効果DSL設計 + 全90枚データ化 | data/effects.json |
| 2 | エンジン実装 + 単体テスト(全カード効果・コンパイル・コントロール) | engine.js, test/ |
| 3 | ソロUI(vs AI はまず合法手ランダム) | auto-play.html |
| 4 | AI強化(ライン評価ヒューリスティック) | engine 内 ai モジュール |
| 5 | オンライン統合(エンジン状態を Firebase 同期、裏向き情報の秘匿) | room-play 改修 |

Phase 0–3 が初回リリース(MVP: ソロで1ゲーム完走できる)。

## 5. リスクと対策

| リスク | 対策 |
|---|---|
| 効果文の裁定が曖昧(タイミング競合・連鎖) | Phase 0 で Codex の裁定を仕様に落とす。不明点は「裁定メモ」に列挙してユーザー確認 |
| 選択UIが複雑化(多段選択・キャンセル) | request/response モデルに統一。エンジンは常に再開可能 |
| オンラインでの裏向きカード秘匿 | Phase 5 で server-authoritative 化を検討(MVP外) |
| 89効果のデータ化ミス | カードごとのエンジンテストで全効果を1本ずつ検証 |
